# 内容权威源与转换流程

## 1. 单一权威源

剧情生产先经过“篇章剧情源”，再进入任务级内容状态：

0. `docs/story/active/`中的剧情母稿是篇章剧情源，用于确定具体剧情事实，但它不等于某个任务已经达到`complete_script`；
1. 任务从`outline`至`complete_script`阶段，Markdown任务剧本是内容权威源，JSON只可作为临时导出，不得手工分叉修改；
2. `data_ready`之后，通过审核的JSON成为运行权威源，审阅版Markdown由JSON自动生成。任何剧情修改先改JSON，再重新生成审阅稿。

篇章剧情源到任务状态的转换必须经过人工拆分和审核，不能把`source_mother_draft`直接映射为`complete_script`。

同一任务不得同时允许编剧改Markdown、开发改JSON。转换时必须记录源文件哈希、转换工具版本和时间。

## 2. 统一内容状态

任务只使用顶层`content_status`，固定状态机：

`outline → design_card → draft_script → complete_script → data_ready → implemented → verified`

不再使用`draft/review/approved`与`writing_status`双轨字段。含义：

- `outline`：只有方向；
- `design_card`：字段齐全但无完整正文；
- `draft_script`：已有正文，仍可缺分支或回访；
- `complete_script`：完整可人工演出；
- `data_ready`：已转成结构化数据并通过静态校验；
- `implemented`：已在游戏中可运行；
- `verified`：自动测试、人工通关和制作方签收通过。

只有`verified`可标记DONE。状态只能向前推进；回退必须在变更记录说明原因。

## 3. 正式数据、示例和测试夹具

- `content/`：正式运行数据，新游戏默认状态必须为未开始；
- `examples/`或本方案中的`samples/`：展示字段形状，仍应使用生产默认值；
- `fixtures/`：测试夹具，可预设任务已完成、背包已满或NPC受伤，但运行时manifest禁止加载；
- `generated/review/`：由JSON生成的审阅稿，不手工编辑。

校验器必须验证运行manifest没有引用`fixtures/`。

## 4. 转换门

`complete_script → data_ready`前必须完成：16类任务字段、完整节点、状态登记、资源引用、失败续接、回访、测试、词表检查和人工通读。转换工具不得通过摘要自动补写剧情，只能把已审核内容转换为数据结构。

## 5. 变更与ID

ID一旦进入`data_ready`不得随意改名。需要重构时保留别名映射和存档迁移。对白文字可修改，但会影响自动快照测试时必须更新批准记录。
