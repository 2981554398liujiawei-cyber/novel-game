# 技术架构

## 1. 技术选型

Godot 4.x稳定版本，GDScript，Windows桌面。内容使用UTF-8 JSON；开发期校验使用Python 3与`jsonschema`。不依赖网络、数据库、第三方Godot插件或运行时大模型。NPC立绘使用透明背景PNG，由Godot导入压缩；运行时不生成图像。

## 2. 模块

- `AppRoot`：应用生命周期、场景切换、全局错误处理。
- `ContentLoader`：读取JSON、Schema验证、跨文件引用检查、构建运行时模型。
- `AssetRegistry`：根据稳定资源ID解析立绘路径、缺图剪影和加载状态。
- `GameState`：唯一状态存储与条件/效果接口，发出状态变化信号。
- `StoryRunner`：执行叙事、对白、选择、检定、奖励、完成节点和跳转，只输出节点事件。
- `DialoguePresenter`：显示姓名牌、对白、动作文本、历史记录和继续状态。
- `PortraitPresenter`：根据`speaker_id`、`expression`和`portrait_action`显示、保持或隐藏立绘，并处理淡入淡出与缺图回退。
- `QuestManager`：任务状态、前置、互斥、目标和回访。
- `CombatRunner`：敏捷行动序、六类动作、冷却/每战次数、同伴倾向、状态效果、撤退、首领阶段、可保存随机种子和结算。
- `InventoryManager`：背包、任务物品、装备、保管箱和商店。
- `RelationshipManager`：信任、好感、边界状态和关系文本版本。
- `WorldClock`：离散时间片与任务事件推进。
- `SaveManager`：槽位、原子写入、备份、迁移和恢复。
- `MainUI`：探索阅读、NPC对话、选择、侧栏、标签页、历史和设置。
- `DebugConsole`：开发命令、状态检查、节点图、当前NPC/表情检查和日志导出。

## 4. 运行流程

启动 → 加载设置 → 载入manifest → Schema验证 → 引用与表情验证 → 构建内容和资源索引 → 读取或新建存档 → 恢复GameState → StoryRunner进入节点 → Dialogue/Portrait Presenter重建当前演出。任何正式内容验证失败都进入错误报告页，不允许带错误继续运行；单张图片加载失败可降级为剪影并继续。

## 5. 信号契约

关键事件使用信号解耦：`story_node_entered`、`dialogue_presented`、`portrait_requested`、`portrait_changed`、`choice_presented`、`state_changed`、`quest_status_changed`、`combat_started`、`combat_finished`、`save_completed`、`content_error`、`asset_warning`。UI只监听信号，不直接修改业务状态。

## 6. 存档原则

存档保存任务节点、游戏状态、随机种子、设置和当前UI模式，不保存纹理对象。读取后通过当前节点和NPC数据重新加载立绘。立绘资源更新不应破坏旧存档；NPC ID和表情ID进入正式内容后保持稳定。

## 7. 自动测试

Python校验负责Schema、重复ID、引用、未知状态、NPC表情和任务图可达性。Godot无界面测试负责条件/效果、战斗确定性、存档往返、迁移、对话模式切换、缺图回退和立绘重建。至少提供一条脚本化流程从新游戏走到`NV_MAIN_003`完成。

## 8. 正式数据与夹具隔离

运行时只加载`content_manifest.json`列出的正式内容，manifest禁止包含`fixtures/`。测试夹具可预设已完成任务或受伤状态，但不能被正式构建扫描。
