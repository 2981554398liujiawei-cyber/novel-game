# 数据与ID规范

## 1. ID

全局资源ID使用大写字母、数字和下划线，例如`NV_MAIN_003`、`NV7_NPC_LANYIN`、`NV7_COMBAT_DUDU_RABBIT`。节点ID使用小写蛇形。进入`data_ready`后ID稳定，改名需要别名与存档迁移。

## 2. 状态键

小写点分路径，例如`quest.nv_main_002.completed_commissions`。每项登记类型、默认值、范围/允许值、持久性、写入来源和读取来源。正式默认值必须代表新游戏；完成态只允许出现在`fixtures/`。需要限制运行时入口时，可用`read_only`禁止外部来源修改，或用`write_sources`登记允许的来源；`system`和`save_restore`属于GameState内部受信来源。

条件与效果使用类型化对象，不使用`A+=2`字符串。数值状态允许`set/inc/dec`，布尔和字符串只允许`set`；条件按类型允许`eq/neq（兼容ne）/gte/lte/gt/lt/in/not_in`。写入越界时拒绝整次操作，不进行静默截断；校验器检查值类型和范围。

## 3. 统一内容状态

任务只使用顶层`content_status`：`outline`、`design_card`、`draft_script`、`complete_script`、`data_ready`、`implemented`、`verified`。不使用`status=approved`或`script.writing_status`。运行时任务进度由状态注册表中的`quest.*.status`表示，两者不得混淆。

## 4. 完整任务结构

任务数据必须结构化包含：基本规格、设计目的、条件与互斥、触发和开场演出、场景表、完整节点、战斗/非战斗机制、分支摘要、失败续接、奖励、世界变化、任务后回访、实现备注和测试。核心对白只存在于节点数据中。

## 5. 演出标签

`gesture`、`delivery`、`camera`、`portrait_action`和音效必须引用`presentation_tags.json`。`portrait_action=keep`仅在所有进入路径上已显示同一说话人时合法；否则必须使用`show`。

## 6. 循环和终止

所有可达路径必须能到达终止节点。允许玩家主动重复的恢复/重试环必须在`allowed_loops`登记节点、最大建议重复次数和退出路径。无出口强连通循环属于阻断错误。

## 7. 正式数据、样例和夹具

`samples/`使用生产默认值；`fixtures/`允许测试预设。正式manifest不能引用夹具。权威源转换见`31A_内容权威源与转换流程.md`，Schema列表见`33_完整数据Schema目录.md`。
