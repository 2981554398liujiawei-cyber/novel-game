# 存档、失败续接与调试规范

## 1. 存档槽

3个手动槽、1个自动槽、1个快速槽。自动保存发生在进入任务、关键选择后、首领战前和任务完成后。存档包含`save_version`、`content_version`、随机种子、当前位置、任务、状态、背包、关系、战斗恢复点和历史摘要。

基础骨架阶段固定槽位为`manual_1`、`manual_2`、`manual_3`、`auto`和`quick`。当前最小存档只序列化`save_version`、`game_version`、创建/更新时间、槽位、游玩时间、StoryRunner故事与节点ID、GameState可持久化快照和可复现随机种子接口；任务、背包、关系、战斗恢复点与历史摘要在对应模块落地后扩展，不由SaveManager提前复制其业务状态。

## 2. 安全写入

先写入`.tmp`，完成JSON解析与校验后替换正式文件；替换前把旧文件保存为`.bak`。读取失败时尝试备份并明确告知玩家。禁止静默重置存档。

基础接口的`load(slot_id)`不会静默改读备份；失败结果会返回错误码及`backup_available`，由调用方明确请求`restore_backup(slot_id)`。恢复前必须在临时运行时中预检GameState快照和StoryRunner目标，再用包含非持久状态的完整运行时检查点提交；任何校验或提交失败都不得改变当前状态、剧情位置、游玩时间或随机状态。读档恢复到完成节点时不得重发剧情完成信号，避免重复结算。

## 3. 迁移

每次更改状态结构必须增加`save_version`并提供迁移函数或明确拒绝加载。删除状态前先在迁移中转换或清理。内容ID一旦进入存档不得直接复用给另一对象。

当前开发期`save_version`为`1`，保留`migrate_save`入口；本阶段不迁移其他版本，而是返回`SAVE_VERSION_UNSUPPORTED`。开发期存档可失效，但不得静默吞掉或重置。

## 4. 失败续接模板

每个可失败任务必须定义：失败触发、玩家立即看到的结果、资源代价、返回地点、任务新状态、NPC新对白、重试入口和不能重复领取的奖励。失败版本仍需通过Schema与可达性检查。

## 5. 调试控制台

仅开发构建启用：跳转节点、设置状态、增加物品、修改关系、开始战斗、切换时间、显示当前任务图、导出日志、重放随机种子。正式私用构建可通过启动参数`--debug-tools`开启，不在默认菜单暴露。

## 6. 日志

错误日志包含时间、版本、任务ID、节点ID、状态摘要和随机种子。玩家可在设置中打开日志目录。日志不记录现实隐私信息。
