# AGENTS.md

本文件是 `novel-game` 仓库级长期执行规则，也是 Codex 在本项目中最常读取的规则入口。任何任务开始前先读本文件；进入 `content/`、`src/`、`tests/`、`docs/spec/`、`docs/story/` 时，再读取距离当前工作目录最近的局部 `AGENTS.md`。高频且不可违反的规则集中在这里，不得依赖制作方每次重复提醒。

## 0. 每次任务开始与结束都必须执行的动作

### 开始前

1. 明确当前任务卡的 **Goal / Context / Constraints / Done when**，不要把“整个游戏”当成一个任务。
2. 读取根 `AGENTS.md`，再读取当前目录最近的局部 `AGENTS.md`。
3. 只读取与任务直接相关的正式规格、剧情源和现有代码；涉及剧情时必须先读取 `docs/story/source_manifest.json`。
4. 搜索现有实现、Schema、状态键和测试，避免重复造一套同名系统。
5. 若发现产品规则、剧情正典或数据来源冲突，先报告 `SPEC_CONFLICT` 或 `STORY_SPEC_CONFLICT`，不要凭个人偏好选一边继续写。

### 完成前

1. 检查实际改动是否超出任务范围。
2. 数据结构变化必须同步 Schema、样例/fixture、校验器、manifest 和相关规格。
3. 功能变化必须同步自动测试。
4. 运行任务要求的验证命令；任何必需验证失败都不得宣称完成。
5. 最终汇报必须列出：修改文件、关键决定、实际执行的验证命令及结果、仍存在的已知限制。

## 1. 最高频不可违背规则

以下规则在所有任务中默认生效：

- **当前只制作第七新手村。** 不得因为剧情包含有后续区域，就提前实现天马城、黑石塔、樱花岛或其他区域。
- **不得自行创作缺失的正式剧情。** 剧情母稿不等于可直接运行的 `complete_script`；缺少完整节点、对白、分支、失败续接或数据字段时报告 `CONTENT_MISSING` 或 `STORY_NOT_DATA_READY`。
- **不得把旧剧情包里的旧产品形态带回项目。** 当前不做自由行走地图、7×5战棋、开放世界实时模拟、后台大规模玩家模拟、联网和运行时AI。
- **正式对白和剧情条件不得硬编码到 GDScript。** 运行时内容必须外置并经 Schema 校验。
- **`GameState` 是业务状态唯一真源。** UI、战斗、任务、关系等模块不得各自维护互相冲突的副本。
- **任何新增状态键先登记。** 禁止在代码或任务数据中临时创造未登记变量。
- **任何新增正式内容先登记 manifest。** 运行时不得自动扫描临时目录、fixture 或剧情源文档。
- **修改数据结构必须同步 Schema 与测试。** 不能只改一端让另一端“以后再说”。
- **缺图、缺音频可安全降级，缺关键剧情不能自动生成。** 资源异常不得卡死游戏；剧情缺口必须显式报错。
- **运行时完全离线。** 禁止 HTTP、WebSocket、云存档、在线登录、遥测、广告、远程配置和运行时大模型调用。
- **不擅自扩大范围、不顺手大重构。** 一个任务解决一个明确目标。
- **代码与正式规格冲突时，不能默认代码正确。** 先报告冲突，再按权威源处理。
- **没有通过 `scripts/validate.ps1` 的内容不得进入正式构建。** 涉及运行时代码还必须执行测试；涉及启动/场景必须执行烟雾测试。

## 2. 项目身份与当前生产边界

- 项目：小说改编的 Windows 单机离线节点式文本 RPG。
- 当前版本范围：第七新手村，从枫月醒来到取得离村资格结束。
- 引擎基线：Godot 4.6.2 Standard。
- 脚本语言：GDScript。
- 目标平台：Windows 10/11 x86-64。
- 基准分辨率：1280×720；主要展示目标：1920×1080。
- 目标时长：3—4.5 小时。
- 玩家可见总文本预算：约 38,000—50,000 中文字。
- 核心循环：阅读/观察 → 选择行动 → 对话、调查、战斗或交易 → 状态变化 → 任务、人物与世界反馈 → 继续探索。
- 当前主界面形态：探索阅读、NPC 对话立绘、文本战斗三种模式。
- 当前不做：自由移动地图、7×5战棋、实时开放世界模拟、多人联网、Live2D、骨骼动画、嘴型、动态换装、配音、环境声分层、动态音乐。

## 3. 指令与权威源优先级

发生冲突时按以下顺序处理：

1. 制作方在当前任务中的明确要求。
2. 当前工作目录最近的局部 `AGENTS.md`。
3. 仓库根目录本文件。
4. `docs/spec/` 的正式产品、世界观、玩法、技术和验收规格。
5. `docs/story/active/` 的当前篇章剧情源，仅负责具体剧情事实、场景和对白来源。
6. `docs/story/roadmap/` 的未来剧情路线草案。
7. `docs/story/reference/` 的历史/未来参考材料。
8. `docs/production/` 的生产流程和任务卡。
9. 现有代码行为。

注意：第4与第5项职责不同。`docs/spec/`决定产品规则和系统边界；`docs/story/active/`决定当前篇章的具体剧情来源。若两者直接冲突，不得简单覆盖，必须报告 `STORY_SPEC_CONFLICT`，并在制作方确认后同步修订两边。

## 4. 当前剧情包的使用规则

仓库中的剧情材料按三层管理：

- `docs/story/active/`：当前第七新手村剧情源。可用于理解故事、拆分任务和准备数据，但只有达到 `complete_script` / `data_ready` 的任务才能进入运行时 JSON。
- `docs/story/roadmap/`：后续区域任务树与长期路线草案。只用于未来规划，**不得改变当前新手村范围**。
- `docs/story/reference/`：樱花岛、旧开放世界设定等参考材料。它们可能包含已废弃的玩法、角色分类和技术要求，只能提供灵感或未来内容线索，不能直接当作当前实现需求。

当前剧情源明确采用：

- 主线001《没有退出键》；
- 主线002《三份委托》：韩石的试刃、苏芷的药篮、顾长川的界石，三项可按任意顺序推进，完成任意两项即可触发主线003，第三项之后仍可补；
- 主线003《兔王不该出现》；
- 主线004《狼影中的同伴》；
- 主线005《悬崖下的宝藏》；
- 主线006《天枢神泉》；
- 主线007《九坛泉水》；
- 主线008《离村之前》。

任何旧文档中出现“黄金铁锤/黄金药鼎/二十条虎尾作为三份委托”“模拟玩家”“7×5战棋”“实时开放世界”等内容，均视为旧草案，不得覆盖上述当前规则。

### 剧情内容状态门

固定流程：

`source_mother_draft → design_card → draft_script → complete_script → data_ready → implemented → verified`

- `source_mother_draft`：剧情母稿，可能缺少16字段和逐节点可执行信息。
- `design_card`：任务设计卡完整，但正文仍不完整。
- `draft_script`：已有场景和对白，但仍允许缺口。
- `complete_script`：可从开场完整演到稳定出口，关键对白、选择、失败续接和反馈齐全。
- `data_ready`：ID、条件、效果、资源引用和测试已冻结，可转运行时 JSON。
- `implemented`：已接入引擎。
- `verified`：自动与人工验收通过。

Codex 不得自行把状态向前提升；每次提升必须满足对应验收并留下记录。

## 5. 正式剧情与内容红线

- **禁止自行创作缺失的正式剧情。**
- 不得根据任务标题、总纲、原著章节索引、未来任务树或几句对白钩子，自行扩写成正式玩家可见剧情。
- 剧情母稿可作为写作源，但不能跳过完整剧本审核直接生成生产 JSON。
- 遇到缺少人物动机、关键对白、任务分支、任务结局、失败续接或世界反馈时，报告 `CONTENT_MISSING`。
- 已有剧情源但尚未达到数据门槛时，报告 `STORY_NOT_DATA_READY`，而不是自动补齐。
- 正式内容不得出现“待补”“待定”“自由发挥”“双方争论一番”“NPC解释情况”“玩家自由选择”“根据需要生成”等占位表达。
- 正式任务必须满足当前剧情剧本规范、Schema、引用、状态、可达性、终止性和测试要求。
- 正式剧情、NPC对白、关键物品说明、任务条件、奖励和世界状态必须来自外部内容数据，不得写死在代码里。

## 6. 世界规则，高频不可改

- 《王者》是索达公司最新推出的游戏。
- 主角现实本名就是“枫月”，进入游戏后继续使用本名。
- 枫月此前没有玩过《王者》，不了解任务、怪物、隐藏剧情和未来事件，只能边走边摸索。不得写成攻略型或预知型主角。
- “冒险者”属于玩家类 NPC：和枫月一样来自现实世界，保留现实记忆，能看到完整系统，无法退出游戏，死亡后可按原著规则复活并承担代价。
- 玩家类 NPC 很少主动对陌生人谈现实，但没有系统禁言。是否谈现实取决于关系、场景和动机。
- 本地居民属于系统 NPC：能看到与自身相关的系统、任务、等级和公告，但不知道索达公司和现实世界。
- 系统 NPC 默认不自动复活。具名系统 NPC 的死亡、离开和关键命运必须遵循原著正典或制作方明确决定。
- 当前新手村无已确认永久死亡节点时，Codex 不得擅自永久杀死具名角色；可使用受伤、暂离、功能替代等版本。
- 魔兽看不到系统，也不能用“血条”“掉率”“任务目标”“玩家界面”等玩家术语思考或说话。
- 不存在“只有枫月来自现实”或“枫月是唯一穿越者”的设定。

## 7. 枫月人格与玩家选择边界

玩家可把枫月演成善良、谨慎、功利、怀疑、冒险、冷淡或有所隐瞒的人。允许私藏资源、要求合理报酬、拒绝帮助、拒绝暧昧、暂时隐瞒情报和选择高风险高收益方案。

禁止提供或暗中执行：

- 主动残虐无辜；
- 性胁迫或强迫亲密；
- 为娱乐折磨 NPC；
- 无动机杀害同伴；
- 对已投降且不再构成威胁的人继续施虐；
- 纯粹为了毁灭村庄或世界的路线。

枫月不是固定圣人，但必须保持人格连续性。重要选择至少改变一项：可玩过程、NPC关系、状态、资源、奖励、经济、后续任务版本或世界反馈。只有一句不同结尾对白不算实质分支。

## 8. 暧昧与关系边界

- 岚音为22岁成年角色。
- 所有关系升温、身体接触和暧昧选择必须建立在清醒、自愿、可拒绝和关系条件满足的前提上。
- 拒绝暧昧不得阻塞主线、降低核心战斗能力或造成系统性惩罚。
- 当前尺度上限：调侃、试探、靠肩、短暂牵手、主动拥抱、含蓄告别。
- 不加入露骨性描写、性胁迫或以身体暴露作为关系升级奖励。
- 关系变化通过态度、回访、选择回应和状态反馈体现，不把角色写成单纯数值奖励。

## 9. 任务结构规则

- 任务结构以当前 `docs/story/active/` 与已同步的 `docs/spec/` 为准，不得为了代码方便改写主要因果关系。
- `NV_MAIN_002《三份委托》`是一个主线阶段，内部包含三项并行委托：
  - 韩石的试刃；
  - 苏芷的药篮；
  - 顾长川的界石。
- 三项委托使用子目标/子状态追踪，不强制伪装成三个独立正式支线。
- 完成任意两项时，`NV_MAIN_002`进入 `qualified` 并允许触发 `NV_MAIN_003`；第三项仍可之后完成；未补完不得卡死离村，但必须改变回访或结算反馈。
- `NV_MAIN_003—008`遵守当前剧情母稿的强因果顺序；可选内容只能在明确开放窗口并行。
- 主线失败优先转化为部分成功、受伤、资源损失、敌人逃脱、世界状态变化、救援版或更困难的后续版本，尽量不强制读档。
- 分支可以汇流，但汇流前必须留下可追踪差异。
- 关键选择后1—2个剧情节点内必须有即时回应，不能让玩家选择像投进无底洞。

## 10. 战斗与检定冻结规则

- 定位：标准 RPG 难度，需要使用技能、消耗品、观察和战术选择。
- 不设置法力值；技能使用冷却回合和每战次数。
- 玩家、同伴和敌人按敏捷决定行动顺序；并列或随机部分使用可保存随机种子。
- 防御持续到角色下一次行动，默认降低40%直接伤害；特殊技能通过数据覆盖，不在代码里散落例外。
- 岚音同行时自动行动；玩家每回合最多设置一次倾向：进攻、防守、支援。
- 普通调查优先“条件满足即成功”；重要检定使用可保存随机种子。
- 结果支持成功、部分成功、失败续接。
- 撤退允许失败；失败消耗行动或产生代价，但不应直接返回主菜单。
- 每个首领只配置1个独特核心机制和2—3个阶段。
- 首次首领战允许战败，理解机制、调整技能和消耗品后应能稳定取胜。
- 战斗规则变化必须同步规格、Schema、数据和自动测试。

## 11. UI、立绘、背景与音频范围

核心主模式只有：

1. 探索阅读模式；
2. NPC对话立绘模式；
3. 文本战斗模式。

人物资产：

- 9名具名NPC使用独立半身立绘；
- 有对白但未具名角色使用3套通用立绘：无名男村民、无名女村民、蒙面冒险者；
- 纯背景人群不进入正式对话，不显示立绘；
- 当前占位资产目标：17张具名角色表情图 + 3张通用人物图 + 1张缺图回退剪影；
- 缺图剪影只用于异常回退，不作为正式路人常规美术。

其他资产：

- 8张简洁地点色调卡；
- 10—15个短音效；
- 3首循环音乐：村庄、野外、首领；
- 不配音、不做环境声分层、不做动态音乐。

布局：

- 最低支持1280×720；
- 低分辨率优先保证正文、角色姓名、玩家选项和战斗关键信息；
- 右侧状态栏可折叠，但主文本与选项不得被立绘遮挡；
- 关键信息不得只依赖颜色表达。

## 12. 架构职责边界

核心模块：

- `AppRoot`：生命周期、启动、全局错误处理；
- `ContentLoader`：加载外部内容、Schema与跨文件引用校验；
- `GameState`：唯一业务状态真源；
- `StoryRunner`：解释剧情节点，不拥有正式剧情正文；
- `QuestManager`：任务状态、前置、互斥、目标与回访；
- `CombatRunner`：回合、AI、状态效果、随机种子与结算；
- `InventoryManager`：背包、装备、任务物品、保管箱与商店；
- `RelationshipManager`：信任、好感、关系阶段与边界；
- `SaveManager`：原子写入、备份、恢复、迁移；
- `MainUI`：显示数据、转发玩家意图，不直接修改业务状态；
- `DebugConsole`：仅开发构建使用。

禁止：

- UI直接写任务、关系、背包或世界状态；
- 多模块各自保存同一业务状态；
- GDScript硬编码正式对白、任务条件、奖励或关键剧情；
- 引入数据库、网络后端、运行时AI或未经批准的Godot插件；
- 为局部任务进行无关的大规模重构；
- 让测试临时代码进入正式运行路径。

## 13. 数据、ID、Schema 与 manifest

- 运行时内容放 `content/`；Schema放 `schemas/`。
- 正式运行只加载 `content/manifest.json`明确登记的文件。
- `fixtures/`、`tests/`、`docs/story/`永远不是运行时内容目录。
- 全局ID一经用于已签收内容即稳定，不随意改名。
- 资源ID：大写字母、数字、下划线，如 `NV7_NPC_001`。
- 任务内部节点ID：小写 snake_case。
- 状态键：小写点分路径，如 `relation.lanyin.trust`。
- 所有状态必须先登记类型、默认值、范围、持久性、读写来源。
- 条件和效果使用类型化对象，不使用 `A+=2`、`time=day` 等表达式字符串。
- 修改数据结构必须同步 Schema、样例/fixture、校验器、规格和测试。
- 新增正式内容文件必须同步 manifest，路径与大小写必须完全一致。

## 14. 剧情到运行时数据的唯一生产流程

1. 剧情源进入 `docs/story/active/`。
2. 按任务拆成16字段完整剧本。
3. 人工审阅并达到 `complete_script`。
4. 冻结ID、状态、资源引用和测试，达到 `data_ready`。
5. 使用转换工具生成 `content/quests/*.json`。
6. 运行Schema、引用、可达性、终止性、状态和资源校验。
7. 从JSON生成审阅稿，避免Markdown与JSON长期双向手改。
8. 接入引擎并完成自动/人工测试后标记 `verified`。

禁止：

- 直接把剧情母稿当生产JSON；
- 同时长期手改Markdown正式剧本和JSON；
- Codex自行补缺口后悄悄标记 `complete_script`；
- 用未来任务树代替当前完整剧本。

## 15. 存档规则

- 原型阶段允许开发存档失效，但必须标记为开发存档。
- 垂直切片首次正式签收后进入存档兼容冻结。
- 兼容冻结后，状态结构变化必须提高 `save_version` 并提供迁移或明确拒绝策略。
- 正式存档使用原子写入并至少保留一份可恢复备份。
- 不允许静默丢档。
- 当前立绘对象不直接序列化；读档后按剧情节点与数据恢复。
- 随机种子必须保存，使关键战斗和重要检定可复现。
- 允许玩家自由手动存档、读取并查看其他选择；关键选择前可自动存档，不设置一周目读档限制。

## 16. 编码与文件约定

- GDScript、场景目录和运行时资源目录使用英文 snake_case。
- 中文规格和剧情文档保留在 `docs/`，不要无意义批量改名。
- 所有文本文件UTF-8。
- 不提交 `.godot/`、构建产物、用户存档、编辑器临时文件、虚拟环境和缓存。
- 新依赖必须说明用途、替代方案和维护风险。
- 优先小而可测试的模块，避免巨型脚本和万能单例。
- 错误必须显式处理；内容校验失败时进入错误报告流程，不允许带错误继续运行。

## 17. 验证命令与完成定义

按任务范围执行：

- `scripts/validate.ps1`：结构、JSON、Schema、引用、manifest、资产、故事源登记和离线约束；
- `scripts/test.ps1`：Python测试和可用时的Godot导入检查；
- `scripts/smoke_test.ps1`：Godot无界面启动；
- `scripts/build_windows.ps1`：Windows导出。

最低完成条件：

1. 任务要求全部实现；
2. 未擅自扩大范围；
3. 相关规格、Schema、测试同步更新；
4. `validate.ps1`通过；
5. 涉及运行时代码时`test.ps1`通过；
6. 涉及启动、场景或构建时`smoke_test.ps1`通过；
7. 涉及发布候选时Windows构建成功；
8. 最终汇报列出修改文件和实际验证结果。

## 18. Git 工作方式

- `main`始终保持可运行、可验证。
- 每个Codex任务使用范围明确的feature分支或独立工作树。
- 一个任务只解决一个清晰目标，不混入无关重构。
- 提交前检查 `git diff`，不得提交缓存、构建物、密钥、用户存档和临时文件。
- 不重写制作方未要求修改的规格或剧情源。
- 不强制推送、不擅自删除历史、不擅自合并 `main`。

## 19. 离线与安全约束

运行时代码禁止：HTTP、WebSocket、远程配置、广告SDK、分析/遥测SDK、云存档、在线登录和运行时AI API。

开发工具和CI可以安装开发依赖，但不得让这些依赖形成游戏运行时网络路径。不得提交API Key、Token、账号密码或私人绝对路径。

## 20. 不明确时的统一处理码

- `CONTENT_MISSING`：正式剧情/对白/结果缺失。
- `STORY_NOT_DATA_READY`：已有剧情源，但未达到可转换运行时数据的门槛。
- `SPEC_CONFLICT`：两份正式产品/技术规格冲突。
- `STORY_SPEC_CONFLICT`：当前剧情源与正式世界观、玩法或范围冲突。
- `DATA_CONTRACT_MISSING`：需要实现但缺少Schema或数据契约。
- `ASSET_MISSING`：资源缺失，可按已定义回退规则继续。

技术实现细节不明确但不影响产品语义时，选择最简单、可测试、可撤销的方案并记录假设；产品规则、世界观、角色命运、任务结局或玩家可见内容不明确时，不得自行决定。
